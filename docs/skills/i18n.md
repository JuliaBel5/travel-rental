# Skill: i18n (RU/EN dictionaries, compile-time sync)

Use this whenever you add or change user-facing text. Wanderstay is bilingual (RU default, EN under `/en`). Russian is the source of truth for the dictionary shape, and TypeScript keeps English in sync at compile time.

## Trigger

Read this when you: add any visible string, rename a locale key, add a new screen, or do responsive work that depends on label length.

## Where locales live

- `src/locales/ru.ts` defines the dictionary object and exports its type.
- `src/locales/en.ts` is typed against that type.
- `src/locales/index.ts` exposes `useTranslation()`, `getDictionary()`, and `localize()`.
- `next.config.ts` registers the locales and routing.

```ts
// next.config.ts
i18n: {
  // Default locale (ru) is served without a path prefix; English lives under /en.
  locales: ["ru", "en"],
  defaultLocale: "ru",
},
```

## The type-sync trap (why builds break on a missing key)

`ru.ts` exports `Dictionary` as its own shape; `en.ts` must satisfy it exactly:

```ts
// src/locales/ru.ts (tail)
export const ru = {
  common: { appName: "Wanderstay" /* ... */ },
  // ...
};
export type Dictionary = typeof ru;
```

```ts
// src/locales/en.ts (head)
import type { Dictionary } from "./ru";

export const en: Dictionary = {
  common: { appName: "Wanderstay" /* ... */ },
  // ...
};
```

Consequence: **add a key to `ru.ts` and `en.ts` in the same change.** A key present in `ru` but missing in `en` (or a typo/rename in either) is a TypeScript error, not a runtime surprise. This is the whole point of the setup; do not work around it by casting `en` to `any`.

## Reading strings

Client components use the hook. It resolves the active locale from the router:

```ts
// src/locales/index.ts
export function useTranslation(): { t: Dictionary; locale: Locale } {
  const { locale } = useRouter();
  const current = toLocale(locale);
  return { t: dictionaries[current], locale: current };
}
```

```tsx
const { t } = useTranslation();
return <h1>{t.auth.signup.title}</h1>;
```

For `getServerSideProps` / `getStaticProps`, use `getDictionary(locale)` instead of the hook.

## Localized content fields (not UI chrome)

Listing titles, descriptions, and review text are stored as `{ ru, en }` JSON (`Localized`). Pick the language with `localize`:

```ts
// src/locales/index.ts
export function localize(value: Localized, locale: string | undefined): string {
  return value[toLocale(locale)];
}
```

## RU-first responsive and formatting

- **Measure layout in RU.** Russian labels are longer than English, so RU is what overflows first. The header proves it: the full nav only appears at `lg` because the RU labels are too wide at `md` (see `docs/skills/ui-components.md`).
- **Pluralization** uses `Intl.PluralRules`, not a ternary, because Russian has three forms. From `src/pages/bookings/index.tsx`:

```tsx
function pluralForm(count, locale, forms: { one: string; few: string; many: string }) {
  const category = new Intl.PluralRules(locale === "ru" ? "ru-RU" : "en-US").select(count);
  if (category === "one") return forms.one;
  if (category === "few") return forms.few;
  return forms.many;
}
```

- **Money** uses `Intl.NumberFormat` via `formatPrice(amount, currency, locale)` in `src/lib/pricing.ts`. Do not hand-format currency.
